02 · LAKEHOUSE ARCHITECTURE — THREE-ZONE MEDALLION

The lakehouse follows a medallion architecture (Bronze → Silver → Gold) using Apache Iceberg tables stored on IBM Cloud Object Storage. Each zone has distinct governance rules, access policies, and retention periods managed through Watsonx.governance.

// WHY ICEBERG ON WATSONX.DATA
Apache Iceberg provides ACID transactions, schema evolution, time-travel queries, and partition evolution — capabilities traditionally available only in expensive data warehouses. On Watsonx.data, Iceberg tables are stored on low-cost object storage but queryable at warehouse speed via Presto, delivering data warehouse performance at data lake prices. This open format also means zero vendor lock-in — the same tables can be read by Spark, Trino, or any Iceberg-compatible engine.
01
BRONZE — Raw Ingestion Zone
Raw data lands here exactly as extracted from source systems. Spark connectors ingest from Oracle (JDBC), Salesforce (REST API), SQL Server (JDBC), IoT (MQTT → Kafka → Spark Streaming), CSV/Excel (S3 drop), and the HR system (SFTP). Data is stored as-is in Iceberg tables with ingestion_timestamp and source_system metadata columns. Retention: 90 days. Access: Data engineers only.
02
SILVER — Cleansed & Conformed Zone
Spark ETL jobs run daily: deduplication, null handling, type standardization, and entity resolution across systems (e.g., matching a Salesforce account to an ERP customer record). A unified customer master and unified product master are created here. Schema enforcement via Iceberg schema evolution — new columns added without breaking downstream queries. Watsonx.governance tags PII columns and tracks data lineage from source to silver table. Retention: 12 months. Access: Data engineers + analysts.
03
GOLD — AI-Ready & Analytics Zone
Business-ready aggregations and feature-engineered datasets for both BI dashboards and AI model training. Includes: customer 360 feature store (purchase frequency, lifetime value, churn risk signals), inventory demand features (seasonal patterns, lead times, stock velocity), and operational KPI cubes. Gold tables are optimized for Presto with partition pruning and column statistics. These datasets connect directly to Watsonx.ai as training data sources for Granite model fine-tuning. Retention: Permanent. Access: All business users via Presto SQL.
03 · GOVERNANCE & WATSONX.AI INTEGRATION

The lakehouse is not just a data store — it is the governed data foundation that makes enterprise AI possible. Without clean, catalogued, access-controlled data, AI models produce unreliable results and create compliance risk.

Data catalogueWatsonx.governance auto-catalogues every Iceberg table with business glossary terms, data steward assignments, and quality scores.
Lineage trackingEnd-to-end lineage from source system → bronze → silver → gold → AI model. Auditors can trace any model prediction back to its raw source records.
PII protectionAutomatic PII detection and column-level masking. Customer names, emails, and phone numbers are tagged and masked for non-privileged users via dynamic data masking rules.

01.1 · ARCHITECTURE DIAGRAM

Fully serverless, event-driven pipeline — a photo upload to S3 triggers the entire verification chain. The Rekognition face collection is pre-indexed during employee onboarding. No polling, no servers, no scaling configuration.

01
Camera captures photo at building entry point
A motion-triggered camera snaps a photo when someone approaches the door and uploads the image to the entry-photos/ prefix in S3.
02
S3 event triggers Lambda automatically
An s3:ObjectCreated event invokes the verify_access Lambda function — zero polling, instant reaction.
03
Lambda calls Rekognition SearchFacesByImage
The function passes the S3 image to Rekognition's SearchFacesByImage API, searching against the employees-authorized face collection with a 95% confidence threshold.
04
Rekognition returns match result with confidence score
If a face matches at ≥95% confidence, Rekognition returns the employee ID (FaceId + ExternalImageId). If no match, it returns an empty result.
05
Access decision: grant or deny
Match → Lambda sends an IoT unlock signal to the door controller and logs "ACCESS_GRANTED". No match → door stays locked, SNS alert sent to security team.
06
Every event logged to DynamoDB audit table
Every access attempt (match or no match) writes a record: timestamp, location, person_id, confidence, match_result, image_key. This creates the immutable SOC 2 audit trail.
07
SNS delivers real-time security alerts
Unrecognized visitors trigger an SNS notification to the security team with the image link and timestamp. After-hours access attempts trigger a separate high-priority alert channel.

02 · EMPLOYEE ENROLLMENT FLOW

During HR onboarding, a new employee's photo is uploaded to S3 and indexed into the Rekognition face collection using the IndexFaces API. The ExternalImageId maps to the employee's HR record, enabling the verification Lambda to return a name and department — not just a FaceId.

01
HR uploads employee photo during onboarding
Photo is stored in employee-photos/{employee_id}.jpg in S3. A separate Lambda function is triggered on upload.
02
Lambda calls Rekognition IndexFaces API
The face is indexed into the employees-authorized collection with ExternalImageId = employee_id. Rekognition stores a mathematical face vector — not the raw image.
03
Employee record updated in DynamoDB
The employee table is updated with face_indexed: true and the Rekognition FaceId. The employee is now ready for facial verification at all entry points.

01.1 · ARCHITECTURE DIAGRAM

Full event-driven pipeline flow — from CSV upload through the S3 Medallion Storage layers to BI dashboards — with numbered steps, supporting services, and data paths shown inline.

// STEP-BY-STEP DATA FLOW
01
Raw CSV file uploaded to Amazon S3
A CSV file lands in the raw/uploads/ prefix. The bucket has versioning enabled, all public access blocked, and data is encrypted at rest with AWS KMS. An s3:ObjectCreated event is automatically emitted — triggering the pipeline.
02
AWS Lambda validates and preprocesses
The S3 event triggers a Lambda function (Python 3.11, 512 MB, 300s timeout). Lambda enforces the schema, checks for nulls, duplicates, and date errors, writes clean rows to curated/year=YYYY/month=MM/ and routes bad rows to quarantine.
03
Lambda failures route to SQS Dead Letter Queue
If Lambda exhausts all retries, the failed invocation is sent to an SQS DLQ. A CloudWatch alarm on DLQ depth fires an SNS email to the data-ops team for forensic investigation.
04
Glue Crawler detects new partitions — ETL job triggered
The AWS Glue Crawler runs every 15 minutes, detects new files in curated/, updates the Glue Data Catalog, then triggers the PySpark ETL job on auto-scaling G.1X workers (2–10). Job Bookmarks prevent reprocessing.
05
Glue ETL transforms data and writes Parquet to Analytics layer
PySpark applies type casting, deduplication (window function), date enrichment (year/month/quarter), and customer segmentation (Platinum/Gold/Silver/Bronze by spend), then writes Parquet to analytics/year=YYYY/month=MM/.
06
Amazon Athena queries Parquet via Glue Catalog
Athena runs serverless SQL directly on Parquet files in analytics/, using the Glue Data Catalog for table definitions and partition metadata. Columnar format reduces scan costs by ~70% vs raw CSV. Pay-per-query pricing.
07
Amazon QuickSight loads results into SPICE
QuickSight connects to Athena and loads data into SPICE (in-memory engine) for sub-second dashboard response. SPICE refreshes incrementally every 15 minutes on transaction_date — near real-time BI at near-zero query cost.
08
QuickSight renders 4 dashboard sheets for business users
Sheet 1 Executive Revenue · Sheet 2 Product Drill-Down · Sheet 3 Customer Intelligence (Bronze→Platinum funnel) · Sheet 4 Pipeline Ops Monitor. Row-level security restricts regional managers to their own data.
09
CloudWatch collects custom metrics and fires alarms
CloudWatch captures Lambda custom metrics: rows_ingested, rows_quarantined, and quarantine_rate. Alarms trigger on threshold breaches or Glue job failures. Lambda X-Ray provides distributed tracing for latency diagnosis.
10
SNS delivers anomaly alerts and pipeline failure notifications
Amazon SNS routes alerts in two cases: (a) a CloudWatch alarm threshold is breached, and (b) QuickSight ML Anomaly Detection flags revenue deviation >2σ from the 30-day rolling baseline. A 30-day forecast with 80% confidence interval displays on the executive sheet.

04 · DATA TRANSFORMATION — AWS GLUE

AWS Glue provides the transformation backbone: Crawlers automatically catalogue incoming schemas, the Data Catalog acts as a central metadata store, and PySpark ETL jobs execute 7 transformation steps to produce analytics-ready Parquet output.

01
Type Casting & Standardization
Cast all columns to correct types. Standardize text fields: customer_id and region normalized to uppercase-trimmed strings.
02
Deduplication via Window Function
Window partitioned by transaction_id, ordered by pipeline_ingest_ts DESC. Keep only the most recent record per transaction — no custom state management needed.
03
Date Dimension Enrichment
Extract year, month, quarter, and week from transaction date. These become Parquet partition columns for fast Athena queries.
04
Customer Segmentation (Platinum / Gold / Silver / Bronze)
Aggregate total spend per customer, then apply tiered segmentation logic. Join segment back to transaction-level data for use in QuickSight filtering.
05
Write Parquet — Partitioned Output
Output partitioned by year/month in Parquet columnar format. Reduces Athena query costs by ~70% vs raw CSV. Job bookmarks prevent reprocessing.
# PySpark ETL Job — key transformation steps from awsglue.context import GlueContext from pyspark.sql import functions as F from pyspark.sql.window import Window # Read from Glue Data Catalog (curated layer) df = glueContext.create_dynamic_frame.from_catalog( database='pipeline_catalog', table_name='curated' ).toDF() # Deduplication via window function window = Window.partitionBy('transaction_id').orderBy(F.desc('pipeline_ingest_ts')) df = df.withColumn('row_num', F.row_number().over(window)).filter(F.col('row_num') == 1) # Date dimension enrichment df = df.withColumn('year', F.year('transaction_date')) \ .withColumn('month', F.month('transaction_date')) \ .withColumn('quarter', F.quarter('transaction_date')) # Customer segmentation customer_spend = df.groupBy('customer_id').agg(F.sum('amount').alias('total_spend')) customer_spend = customer_spend.withColumn('segment', F.when(F.col('total_spend') >= 10000, 'Platinum') .when(F.col('total_spend') >= 5000, 'Gold') .when(F.col('total_spend') >= 1000, 'Silver') .otherwise('Bronze')) # Write Parquet partitioned by year/month df.write.mode('overwrite').partitionBy('year','month').parquet(output_path)

01 · PROJECT OVERVIEW

A mid-size bank processes over 500,000 transactions daily. Regulators require the institution to detect and report suspicious activity within 30 days under the Bank Secrecy Act (BSA). The legacy on-premises AML system misses pattern-based fraud and generates hundreds of false positives daily, overloading compliance analysts.

This project builds an AWS-native Anti-Money Laundering pipeline orchestrated by Apache Airflow (MWAA). Transactions stream in via Kinesis, are enriched and scored by a SageMaker ML model, and suspicious cases are automatically routed to compliance analysts with a pre-populated SAR (Suspicious Activity Report). Airflow DAGs coordinate every stage with retry logic, SLA monitoring, and full audit trails.

This pipeline automates the detection, documentation, and escalation workflow end-to-end — reducing analyst workload while strengthening regulatory defensibility.

02 · ARCHITECTURE — AIRFLOW-ORCHESTRATED PIPELINE

Five stages orchestrated end-to-end by Apache Airflow (MWAA): ingest → enrich → score → alert → report. Each stage is a DAG task with retry logic, SLA deadlines, and CloudWatch observability.

03 · HOW IT WORKS — AIRFLOW DAG STAGES
01
Transactions stream into Kinesis Data Streams
Bank transaction events (card, wire, ACH) are published to a Kinesis Data Stream in real time. Kinesis Firehose buffers and delivers raw JSON records to the S3 raw/transactions/ prefix every 60 seconds. Each record includes transaction ID, amount, timestamp, account IDs, merchant category, and IP metadata. Data is encrypted in transit and at rest with KMS.
02
Airflow DAG triggers — enrichment stage begins
An Airflow DAG scheduled every 15 minutes detects new S3 partitions and kicks off the pipeline. The first task runs an AWS Glue PySpark job that normalizes transaction records, joins against a customer profile table in RDS (account age, average spend, geography), and computes behavioral features: velocity (txns in last 1h/24h), deviation from average spend, and cross-border flag. Enriched records land in processed/enriched/ as Parquet.
03
SageMaker scores every transaction for AML risk
The next DAG task invokes a SageMaker batch transform job on the enriched Parquet file. An XGBoost model trained on the IEEE-CIS Fraud Detection dataset outputs a risk_score between 0 and 1 for each transaction. Features include transaction amount, velocity, merchant category, device fingerprint, and customer behavioral baseline. Scores are written back to S3 and loaded into Redshift via COPY command.
04
Lambda routes high-risk transactions to compliance analysts
A DAG task triggers a Lambda function that queries Redshift for transactions where risk_score > 0.75. For each flagged transaction, Lambda auto-populates a SAR draft with account details, transaction narrative, risk factors, and supporting evidence, then publishes an SNS notification to the compliance team's queue with a direct link to the case. Analysts receive a complete alert within 90 seconds of the original transaction.
05
Redshift + QuickSight deliver compliance dashboards and SAR reports
Redshift stores the full transaction history, risk scores, alert dispositions, and SAR status. QuickSight surfaces a compliance dashboard showing daily alert volumes, false positive rates, high-risk customer segments, and open SAR pipeline. A daily automated PDF report is generated and emailed to the Chief Compliance Officer every morning at 7:00 AM.
06
Airflow monitors SLAs and CloudWatch tracks security
Airflow SLA Miss callbacks fire if any DAG stage exceeds its time budget — enrichment must complete in 8 min, scoring in 5 min, alert dispatch in 2 min. CloudWatch monitors Kinesis consumer lag, Glue job failures, and Lambda error rates. GuardDuty monitors for unauthorized access to transaction data, and findings trigger an SNS alert to the security team immediately.
